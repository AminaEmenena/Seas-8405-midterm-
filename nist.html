NIST SP 800-207 & CISA Zero Trust Maturity Model

Comprehensive Guide to Zero Trust Architecture and Implementation

1. Zero Trust Architecture — Core Principle
  • "Never trust, always verify" — NIST SP 800-207
  • Assumes no implicit trust based on network location. Every access request is fully authenticated, authorized, and encrypted, regardless of where it originates.
  • No Trusted Internal Network: traditional perimeter security assumed inside = safe. Zero Trust recognizes breaches happen and threats exist INSIDE the perimeter.
  • Designed to prevent breaches and limit lateral movement. If attacker gains initial access, they can't move freely.
  • Continuous Verification: trust is never assumed, always verified. Every access decision considers: Who (identity), What (resource), Where (location), When (time), How (device posture).
2. NIST 800-207 Seven Tenets
  1. All data sources and computing services are considered resources
  2. All communication is secured regardless of network location
  3. Access to individual enterprise resources is granted on a per-session basis
  4. Access to resources is determined by dynamic policy (including client identity, application, observable state of requesting asset, behavioral/environmental attributes)
  5. Enterprise monitors and measures integrity and security posture of all owned/associated assets
  6. All resource authentication and authorization are dynamic and strictly enforced before access is allowed
  7. Enterprise collects as much information as possible about current state of assets, network infrastructure, and communications to improve security posture
3. Zero Trust Logical Components
  • Policy Engine (PE): Makes the ultimate decision to grant or deny access. "Brain of Zero Trust." Evaluates policies, risk, threat intelligence. Ex: user normally accesses 5 files/day, suddenly accesses 500 → PE elevates risk score. Is the IP known? Device known? Check logs.
  • Policy Administrator (PA): "The enforcer." Takes PE decision and makes it happen. Establishes or shuts down communication paths between subjects and resources based on PE decisions.
  • Policy Enforcement Point (PEP): "The gateway/bouncer." Enables, monitors, and terminates connections. Enforces PE decisions in real-time. Sits between user and resources, checks IDs, enforces rules. ALL traffic goes through PEP.
  • SIEM Integration: Collects security events for analysis. Provides data for policy decisions and threat detection. Feedback loop — tracks all successful and failed authentications, user behavior, network patterns.
  • Examples: Impossible travel detection (login from NY at 10am, then London at 10:15am → PE denies London login). Behavior analytics (usually 5 files/day, suddenly 500 → PE sees elevated risk, denies access or requires additional auth).
4. Trust Algorithm

How the PE makes decisions. Considers multiple inputs:

  • User/subject identity and credentials
  • Device health and compliance status
  • Resource sensitivity level
  • Behavioral patterns and anomalies
  • Threat intelligence feeds
  • Time and location context
  • Dynamic: continuously re-evaluated, not just at initial login
5. CISA Zero Trust Maturity Model
  • 5 Pillars: Identity, Devices, Networks, Applications & Workloads, Data
  • 4 Maturity Levels: Traditional → Initial → Advanced → Optimal
  • 3 Cross-Cutting Capabilities: Visibility & Analytics, Automation & Orchestration, Governance
6. Five Pillars Deep Dive
  • Identity: verification of who/what is accessing resources. MFA, identity governance, continuous validation. Move from "traditional" (limited MFA) to "optimal" (continuous verification, risk-based access).
  • Devices: assessment of device health and compliance. Endpoint detection, device inventory, health attestation. Traditional (limited visibility) → Optimal (real-time compliance, automated remediation).
  • Networks: segmentation and monitoring. Micro-segmentation, encrypted traffic, network monitoring. Traditional (perimeter-based) → Optimal (fully micro-segmented, encrypted, monitored).
  • Applications & Workloads: secure application delivery. App security testing, secure development, workload protection. Traditional (limited testing) → Optimal (continuous testing, secure by design).
  • Data: protection of data assets. Classification, encryption, DLP, access controls. Traditional (limited classification) → Optimal (automated classification, granular controls).
7. Maturity Levels Explained
  • Traditional: manually configured, static policies, limited visibility. Perimeter-based thinking still dominates.
  • Initial: some automation, beginning to implement ZT principles. Starting MFA rollout, basic device management.
  • Advanced: centralized identity, automated provisioning, broad MFA. Micro-segmentation beginning. Policy enforcement becoming dynamic.
  • Optimal: fully automated, continuous verification, adaptive policies. Real-time risk assessment. All pillars integrated and operating at highest maturity.
8. Zero Trust Deployment Approaches
  • Enhanced Identity Governance: focus on identity as primary security control. Strong authentication, access policies based on identity attributes. Good starting point for most organizations.
  • Micro-Segmentation: divide network into small, isolated segments. Each segment has own security controls. Limits lateral movement.
  • Software Defined Perimeter (SDP): creates dynamic, individualized network boundaries around each user/device/application. Resources invisible to unauthorized users.
9. Zero Trust Implementation — Practical Considerations
  • It's a journey, not a destination. Phased approach recommended.
  • Start with identity (MFA, SSO) as foundation
  • Expand to device health monitoring
  • Implement network segmentation progressively
  • Add application-layer controls
  • Data classification and protection as ongoing effort
  • Measure progress against CISA maturity model
