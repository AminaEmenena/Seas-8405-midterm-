1. Requirements Engineering 8-Step Lifecycle

The requirements engineering lifecycle guides how we gather, document, and manage security and functional requirements across a system's lifetime:

Step Description
Elicit Gather requirements from stakeholders through interviews, workshops, surveys, document analysis, and observation
Document Record requirements in clear, measurable format (SMART framework) with traceability IDs
Review Validate requirements with stakeholders; ensure clarity, completeness, and feasibility
Communicate Share requirements across teams; ensure engineering, security, and business alignment
Implement Build system features and controls that satisfy documented requirements
Validate Test that implementation meets acceptance criteria; verify security controls work as intended
Monitor Track requirement effectiveness in production (MTTD, incident response, compliance audits)
Update Refine requirements based on new threats, regulatory changes, and operational lessons learned

Requirements Elicitation Techniques

  • Interviews — One-on-one with key stakeholders (security team, compliance, business owners) to understand their pain points and priorities
  • Workshops — Cross-functional group sessions to align conflicting viewpoints and build consensus
  • Surveys & Questionnaires — Gather feedback from large populations; efficient for scaling input
  • Document Analysis — Review existing standards and frameworks (ISO 27001, NIST Cybersecurity Framework, PCI-DSS) to identify gaps and compliance requirements
  • Observation & Use Case Analysis — Watch users in real workflows to uncover implicit requirements and edge cases
Key Insight: Different stakeholders have fundamentally different requirements:
  • Business: Wants performance, scalability, cost-effectiveness. "How fast can we move?"
  • Security: Wants control, visibility, threat detection. "What can go wrong?"
  • Users: Want usability, speed, minimal friction. "Is this easy to use?"
Good elicitation balances all three perspectives rather than optimizing one at the expense of others.
2. Functional vs Non-Functional Requirements

Functional Requirements (FR)

WHAT the system does.

  • Authentication (MFA, password reset)
  • Authorization (role-based access)
  • Encryption (data at rest/transit)
  • Audit Logging (track who did what)
  • Threat Detection & Incident Response

User-facing, easy to test directly.

Non-Functional Requirements (NFR)

HOW WELL the system performs.

  • Response time, throughput
  • Availability, reliability
  • Scalability, maintainability
  • Usability, portability
  • Testability, security properties

Harder to test, often architectural.

NFR Deep Dive: Categories & Examples

NFR Category Definition Example
Capability Performance metrics: response time, throughput, latency MFA must complete in under 5 seconds 99.9% of the time
Scalability System grows with demand: horizontal (add servers) vs vertical (bigger hardware) Support 100,000 concurrent users. Target website crash = scalability failure under load
Availability Uptime percentage. Higher percentages = lower acceptable downtime 99.95% = 22 min/month | 99.9% = 43 min | 99.99% = 4.2 min | 99.999% = 26 sec
Reliability MTBF (Mean Time Between Failures), MTTR (Mean Time To Recover) If org has zero failure rotations in 6 months and doesn't notice, security control isn't actively monitored
Maintainability How quickly can we patch and respond to new threats? Security patches within 24 hours. Zero-day exploit = immediate response required
Usability Can users actually use security controls correctly? If MFA is too complex, users bypass it. "A secure system no one uses correctly is useless"
Portability Works across platforms and environments consistently Users shouldn't have different security experiences on desktop vs mobile vs cloud
Testability How easily can we validate security behavior? Automated security tests, CI/CD integration, pen test harness
KEY Architectural Impact: Non-functional requirements generally drive architecture MORE than functional requirements do. A system might have identical FR (authentication, encryption) but completely different architectures based on NFR (5 vs 100,000 concurrent users requires different database design, caching strategy, etc.)
3. SMART Framework for Requirements

SMART is a framework for writing clear, measurable, actionable requirements:

Component Definition Example
Specific Clearly defines what is required, not vague "Implement MFA" ✓ vs "Make system more secure" ✗
Measurable Includes quantifiable criteria for success "MFA success rate ≥99.9%" ✓ vs "MFA should work" ✗
Achievable Realistic given technical and resource constraints "<5 sec response time" ✓ vs "<100ms" (unrealistic for distributed system) ✗
Relevant Aligns with business objectives and risk priorities "MFA for VPN" (prevents 80% of breaches) ✓ vs "MFA for system clock API" ✗
Time-bound Specifies when implementation is required "Deploy by Q2 2026" ✓ vs "eventually" ✗

Documentation Examples

Functional Requirement Template

  • ID: FR-001
  • Description: User authentication via password and MFA
  • Acceptance Criteria: 99.9% of login attempts succeed; failed auth logged within 1 second
  • Priority: High (blocks all other features)
  • Stakeholder: Security Team, DevOps, Product

Non-Functional Requirement Template

  • ID: NFR-005
  • Metric: Response time
  • Target: ≤2 seconds for 95th percentile
  • Measurement Method: Load testing, APM monitoring, synthetic checks
  • Priority: Medium (user experience)
4. Requirements Documentation

Clear, measurable documentation ensures all stakeholders (engineering, security, business, compliance) share the same understanding.

Why Documentation Matters

  • Accurate Estimation: Engineering can size the effort when requirements are specific
  • Facilitates Traceability: Link requirements to tests, architecture decisions, and compliance controls
  • Objective Acceptance Criteria: No ambiguity about what "done" means
  • Stakeholder Alignment: Written down = less "I thought we agreed on..." later
  • Compliance Evidence: Audit trail shows we met regulatory requirements intentionally
Real Impact: Requirements without numbers lead to miscommunication. "The system should be fast" means something different to everyone. "≤2 second response time 95% of the time, measured by APM" = everyone agrees on what success looks like.
5. 8 Common Pitfalls in Requirements Definition
Pitfall Problem Fix
Vague Requirements "The system should be secure" — meaningless without specifics Define exactly what "secure" means: data encrypted, auth required, audit logging enabled
Missing NFR Focus only on features, ignore performance/scalability/reliability Explicitly list availability target, response time, concurrent user count
Ignoring Trade-offs Want everything: security + speed + simplicity + low cost Document trade-offs explicitly; make conscious choices
No Traceability Requirement disappears into code; can't prove compliance Link FR/NFR → architecture decision → test case → compliance control
Forgetting Stakeholders Only listen to engineering; miss business and security concerns Include security team, compliance, product, users in elicitation
Over-Specifying Too detailed requirements constrain design unnecessarily Specify WHAT but not HOW; let engineers choose implementation
Not Validating Assumptions aren't verified; build wrong thing Review requirements with stakeholders before coding starts
Not Updating Requirements don't change as threats/regulations evolve Review and update requirements quarterly or after major incidents
6. Diversity vs Redundancy in Architecture

Redundancy

Multiple instances of the SAME resource.

  • Key word: "same"
  • Like having two identical spare tires
  • Examples:
    • Two identical firewalls (Cisco + Cisco)
    • Redundant power supplies
    • RAID arrays
    • Database replicas
  • When: Hardware failures, power outages
  • Benefit: If one fails, another takes over seamlessly

Diversity

Different technologies/vendors for the same function.

  • Key word: "different"
  • Like having a spare tire + run-flat tire
  • Examples:
    • Two firewalls from different vendors (Cisco + Palo Alto)
    • Multiple authentication methods (password + FIDO2)
    • Different encryption algorithms
    • Multi-cloud deployment
  • When: Zero-day vulnerabilities, vendor-specific exploits
  • Benefit: Vulnerability in one doesn't compromise entire system
KEY: Critical systems need BOTH.
  • Redundancy handles individual component failures (hardware, power)
  • Diversity handles systematic failures (zero-days, vendor exploits)
A bank with 2 identical firewalls from the same vendor is vulnerable to a single zero-day affecting that vendor. A bank with diverse firewalls has a better chance of containing impact.
Healthcare Case Study: Requirements in Practice

Scenario: Regional healthcare provider with 50,000 patients, 500 daily logins across clinical systems, and strict HIPAA/HITECH compliance mandates.

Requirements Derived from Context

CategoryRequirementDriver
Authentication MFA for all clinical system access; badge-tap + PIN for workstation login HIPAA §164.312(d) — person/entity authentication
Authorization RBAC with clinical roles (nurse, physician, admin); minimum necessary access to PHI HIPAA Minimum Necessary Rule
Audit Logging All PHI access logged with user, timestamp, patient ID, action; 6-year retention HIPAA §164.312(b) — audit controls
Availability 99.99% uptime for EHR; 15-second max response time during peak hours (500 concurrent users) Patient safety — downtime = delayed care
Encryption AES-256 at rest, TLS 1.3 in transit for all PHI HIPAA §164.312(a)(2)(iv) — encryption standard
Why This Case Matters: It shows how a real-world context (patient count, login volume, regulation) directly drives specific, measurable requirements. "50k patients" isn't just a number — it determines your database sizing, backup strategy, and breach notification scope (HIPAA requires notifying HHS if 500+ records are affected).
7. MITRE ATT&CK in Context (Review)

MITRE ATT&CK provides a structured, community-driven knowledge base of adversary tactics and techniques based on real-world observations.

STRIDE Threat Model (TM)

Threat Definition Example
Spoofing Identity fraud — claiming to be someone else Attacker forges auth token, impersonates admin
Tampering Unauthorized modification of data/system Modify transaction in flight, corrupt audit log
Repudiation Deny responsibility for action User claims they didn't authorize transaction (no audit trail)
Information Disclosure Unauthorized access to sensitive data Read database without permission, memory leak
Denial of Service Make system unavailable to legitimate users DDoS attack, resource exhaustion
Elevation of Privilege Gain higher permissions than authorized User escalates to admin, bypass RBAC check

Risk Framework

Risk = Threat × Vulnerability × Impact

  • Threat: Adversary intent and capability (from ATT&CK)
  • Vulnerability: Weakness in system we can fix
  • Impact: Damage if threat exploits vulnerability (financial, compliance, reputation)

Risk Response Options

  • Accept: Tolerate the risk; document decision and monitoring
  • Mitigate: Reduce likelihood or impact (implement controls)
  • Transfer: Shift risk to third party (insurance, outsourcing, SaaS)
  • Avoid: Don't engage in risky activity (e.g., don't store password in plaintext, don't build certain features)

Requirements-to-MITRE Mapping Chain

  1. Threat Analysis: Use STRIDE/MITRE ATT&CK to identify threats
  2. Control ID: Assign control from framework (e.g., "MFA-001")
  3. Functional Requirement: Define what control does (FR: "Users must authenticate with password + TOTP")
  4. Non-Functional Requirement: Define how well it must perform (NFR: "Auth completes in <5 sec")
  5. Architecture Decision: How to implement (Database + Authenticator service + Audit logging)
8. Requirements Prioritization

Impact vs Effort Matrix (4 Quadrants)

Quadrant Characteristics Action Example
High Impact / Low Effort Quick Wins — big benefit, small effort Do first, highest ROI Enable event logging, update weak passwords
High Impact / High Effort Major Projects — critical but resource-intensive Plan timeline, secure budget, break into phases Zero-trust architecture, cloud migration
Low Impact / Low Effort Nice-to-Have — minimal effort, minimal benefit Do if time permits, fill gaps Security dashboard, custom reporting
Low Impact / High Effort Distractions — waste of time and resources Question if needed, deprioritize, decline MFA for 1% of users, build custom SIEM

Prioritization Criteria (in order)

  1. Business Impact: Does it solve a critical problem?
  2. Risk: Does it reduce high-likelihood or high-impact threats?
  3. Regulatory Requirements: Is it legally mandated?
  4. Technical Feasibility: Can we actually build it?
  5. Cost: Is it worth the ROI?

Example: MFA Prioritization

Good Communication (prioritize):
  • Prevents 80% of documented breach techniques (Business Impact)
  • Required by compliance frameworks (Regulatory)
  • Implementation feasible, 8-week timeline (Feasibility)
  • Cost-benefit: $500k investment prevents $50M+ breach (Cost)
Bad Communication (deprioritize):
  • "Because I said so."
  • "Everyone else has MFA."
  • "It's best practice."
9. Trade-offs in Architecture

Every architecture decision involves trade-offs. The job of an architect is to make them explicitly and document the reasoning.

Trade-off Tension Resolution
Performance vs Security Encryption is CPU-heavy; deep packet inspection slows bandwidth Use hardware acceleration, optimize cipher suites, measure performance impact
Scalability vs Simplicity 1k users = simple architecture works; 1M users = need distribution, caching, sharding Right-size architecture to expected growth; plan migration path
Compliance vs Innovation 90-day security review may slow AI/ML deployment Risk-based review; fast-track low-risk features; pause high-risk ones
Security vs Usability Complex MFA = users bypass it; simple = vulnerable Implement usable security (FIDO2 > SMS); user testing before deployment
Availability vs Cost Multi-region redundancy = expensive; single region = faster failure Match availability target to business criticality; use incremental approach
Architect's Perspective: "Our job is to document trade-offs explicitly and make informed choices, not to satisfy everyone. Here's how we balance:
  1. Identify Trade-off: What are we trading? (security vs speed, cost vs reliability?)
  2. Involve Stakeholders: What does business care about? Security team? Ops?
  3. Technical Constraints: What are hard limits? (budget, time, tech stack)
  4. Cost-Benefit Analysis: What's the business impact of each choice?
  5. Document Decision: Record why we chose A over B, risks accepted
"
10. Risk Management

ISO 31000:2018 Risk Management Process

Step Description Output
1. Establish Context Understand org mission, risk appetite, stakeholders, external environment Risk appetite statement, stakeholder map
2. Risk Identification Use threat modeling (STRIDE, ATT&CK) to find threats Risk register, threat list
3. Risk Analysis Estimate likelihood × impact for each risk Risk ratings (critical, high, medium, low)
4. Risk Evaluation Compare against risk appetite; decide which risks matter Prioritized risk list
5. Risk Treatment Implement controls (mitigate, accept, transfer, avoid) Control plan, implementation roadmap
6. Monitor & Review Track control effectiveness (MTTD, incident metrics); update as needed Metrics dashboard, quarterly reviews

Risk Appetite by Industry

Low Appetite

  • Financial Services: Heavily regulated, lots at stake
  • Healthcare: Patient safety paramount
  • Government/Military: Classified data, national security

Default: Mitigate first, ask questions later

Higher Appetite

  • Startups: Speed-to-market matters; accept risk to move fast
  • Consumer Tech: Innovation often outweighs security

Default: Move fast, mitigate critical risks only

KEY: Architecture must match organization's risk appetite, not generic "best practices." A startup deploying an MVP doesn't need the same infrastructure investment as a bank. Document the appetite and design accordingly.

MTTD: The Measurement Problem

Industry Standard MTTD (Mean Time To Detect): 200+ days!
  • Many orgs don't know they're breached for months
  • "Without measurement we can't improve"
  • Goals: Reduce MTTD through better logging, monitoring, alerting
