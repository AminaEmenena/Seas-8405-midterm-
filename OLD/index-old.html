CIA Triad▶
The Three Pillars
C = Confidentiality · I = Integrity · A = Availability

Definitions

  • Confidentiality: Info only disclosed to authorized people. Data is confidential unless you have a business need to know.
  • Integrity: Info is reliable and cannot be changed unless you're authorized to change it.
  • Availability: Resources can be accessed by authorized users when needed without disruption.
KEY: These three are complementary. They work together to provide a balanced security posture across people, processes, and technology. Network security architecture must ensure all three are maintained.

WEEK 2 — Risk Management & MITRE Frameworks ▶
Risk Management Fundamentals▶

Risk = Threat × Vulnerability × Impact

  • Threat: Who or what could cause harm (hackers, natural disasters, insiders)
  • Vulnerability: A weakness that can be exploited
  • Impact: The damage if the threat exploits the vulnerability

Risk Response Strategies

StrategyWhat You DoExample
AcceptAcknowledge and live with itLow-impact legacy system risk
MitigateReduce likelihood or impactAdd firewall, train employees
TransferShift risk to another partyBuy cyber insurance
AvoidEliminate the activity entirelyDon't store data you don't need
MITRE ATT&CK Framework (Offense)▶
ATT&CK = Adversarial Tactics, Techniques & Common Knowledge — describes HOW attackers operate. Think of it as the attacker's playbook.

Core Structure

  • Tactics: The attacker's GOALS (the WHY) — e.g., Initial Access, Lateral Movement, Exfiltration
  • Techniques: HOW they achieve each goal — e.g., Phishing, Pass-the-Hash
  • Procedures: Specific implementations by real threat groups

ATT&CK Use Cases

  • Threat Detection & Hunting: Map your detections to ATT&CK techniques to find gaps
  • Red Teaming: Simulate real attacker behaviors using ATT&CK TTPs
  • Gap Analysis: Compare your controls against known techniques to find blind spots
MITRE D3FEND Framework (Defense)▶
D3FEND is the defensive counterpart to ATT&CK. It maps defensive techniques to counter offensive ones.
D3FEND Categories
Model → Harden → Detect → Isolate → Deceive → Evict

The Six D3FEND Categories

CategoryWhat It Does

HardenStrengthen systems against attack
DetectIdentify malicious activity
IsolateContain threats and limit blast radius
DeceiveMislead attackers (honeypots, decoys)
EvictRemove the attacker from your environment
MITRE Engage (Active Defense)▶

MITRE Engage — Three Strategic Goals

How the MITRE frameworks fit together: ATT&CK maps offensive techniques → D3FEND maps defensive countermeasures → Engage provides active defense strategy. Together they form a complete ecosystem.

WEEK 3 — Network Security Architecture ▶
CIA Triad in Network Architecture▶

Confidentiality in Networks

  • Encrypted Protocols: SSH, TLS, IPsec, S-MAIL for sensitive data
  • Network Segmentation: VLANs and zones where only authorized parties access traffic
  • Secure Tunneling: Tunnel insecure ports over secure protocols to prevent eavesdropping
  • Access Controls: Strong authentication and authorization for network resources

Integrity in Networks

  • Transmission Integrity: Checksums and cryptographic verification to ensure data isn't altered in transit
  • Protocol Validation: Verify data hasn't been tampered with between sender and receiver

Availability in Networks

  • Redundant paths, failover systems, DDoS protection
  • Load balancing, fault tolerance, disaster recovery

Scope, Decision-Making & Build vs Buy▶
Scope is Iterative, Not Fixed. It changes as result of organizational changes, design decisions, or deliberately by the architecture team. Four steps: Initial Scope Setting → Requirements Gathering → Design Development → Scope Documentation.

Critical Architectural Risks & Empty Scope

  • Sometimes constraints are so tight that the architect's scope is effectively zero
  • Recognize warning signs: minimal change possible due to constraints, tech debt, or org barriers
  • Address root causes first: clear hurdles before attempting architecture with hands tied
  • Use initial scope as a diagnostic: identify blockers early, work with execs to clear the path

Cloud Security Challenges▶

Top Cloud Security Risks

  • Misconfiguration: #1 cause of cloud breaches — open S3 buckets, permissive security groups
  • Identity & Access: Overprivileged accounts, poor credential management
  • Data Exposure: Inadequate encryption, improper data classification
  • Shared Technology: Multi-tenancy risks — side-channel attacks, resource competition
  • Insecure APIs: Poorly secured cloud management APIs can be exploited
  • Lack of Visibility: Hard to see what's happening in someone else's infrastructure

Cloud Security Controls

  • CASB (Cloud Access Security Broker): Sits between users and cloud services, enforces security policies
  • CSPM (Cloud Security Posture Management): Scans for misconfigurations across cloud accounts
  • CWPP (Cloud Workload Protection Platform): Protects workloads running in cloud environments
  • Encryption: Data at rest AND in transit — customer-managed keys when possible

WEEK 6 — Zero Trust Architecture ▶
Zero Trust Core Principles▶
The Zero Trust Motto
"Never Trust, Always Verify"
Key shift: Traditional security = trust everything inside the network perimeter. Zero Trust = trust NOTHING regardless of location. Every access request must be verified.

NIST SP 800-207 — Zero Trust Tenets

  • All data sources and computing services are considered resources
  • All communication is secured regardless of network location
  • Access is granted on a per-session basis
  • Access is determined by dynamic policy — including user identity, device health, behavior, environment
  • The enterprise monitors and measures the security posture of all assets
  • Authentication and authorization are strictly enforced before access
  • The enterprise collects information about assets and uses it to improve security posture
Zero Trust Logical Components (NIST 800-207)▶

Core Components

ComponentRole
Policy Engine (PE)The "brain" — makes access decisions based on policy
Policy Administrator (PA)Executes the PE's decision — establishes or shuts down communication paths
Policy Enforcement Point (PEP)The "gate" — enables, monitors, and terminates connections
Subject → [PEP] ←→ [PA] ←→ [PE] ←→ [Trust Algorithm] ↑ Uses: identity, device health, behavior, threat intel, policies

Trust Algorithm Inputs

  • User identity and credentials
  • Device health and compliance status
  • Behavioral patterns and anomalies
  • Threat intelligence feeds
  • Environmental context (time, location, network)

Incident Response Process▶
NIST Incident Response Lifecycle
Prepare → Detect & Analyze → Contain, Eradicate, Recover → Post-Incident

The Four Phases

PhaseKey Activities
1. PreparationBuild the team, create playbooks, set up tools, train staff, conduct exercises
2. Detection & AnalysisMonitor for indicators of compromise (IoCs), triage alerts, determine scope and severity
3. Containment, Eradication, RecoveryStop the bleeding (contain), remove the threat (eradicate), restore systems (recover)
4. Post-Incident ActivityLessons learned, root cause analysis, update playbooks, improve defenses
Metrics that matter: MTTD (Mean Time to Detect) — how fast you spot threats. MTTR (Mean Time to Respond/Recover) — how fast you fix them. Lower is better for both.

Case Study 2: Cimpress/Vistaprint Zero Trust▶

  • Needed to move from perimeter-based security to Zero Trust

Phased Implementation Approach

PhaseFocus
Phase 1MFA rollout across the organization — quick security win
Phase 2Zero Trust authentication broker — device-first approach, device management
Phase 3EDR deployment and continuous monitoring
Key Results: MTTD decreased from MONTHS to HOURS. Business continuity maintained during COVID transition to remote work. Device-first approach was critical — focused on device trust before user trust.

Key Lessons

  • Start with MFA — it's a quick, high-impact win
  • Use a phased approach, not big-bang
  • Device trust is as important as user trust
  • Zero Trust enabled seamless remote work during COVID
