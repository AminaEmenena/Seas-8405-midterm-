📱 1. Platform Topics — BYOD, COPE & Containerization
KEY IDEA: One of the biggest challenges in cybersecurity is managing personal devices that access corporate data. The mixing of sensitive corporate data with personal data on employee-owned devices has never been properly solved.

Organizations have tried several models for managing mobile devices, each with trade-offs between employee convenience and security control. The government in particular is moving away from employee-owned models due to persistent security issues.

Device Ownership Models

ModelWho OwnsWho ControlsStatus
BYODEmployeeEmployee (some MDM)Never had a good policy. "Hanging on by thin thread."
POCEEmployeeCorporate tools (MDM)Government getting away from this entirely
COPECompanyCompany (allows personal use)Where more companies are going — more control
Professor Duke: "There was never a good BYOD policy to start with. Everybody thought it was going to make things more efficient and cost effective — it was a bunch of garbage. COPE works better because the company has more power in terms of policing and protecting devices."

Containerization (Slide 5)

Definition: Containerization is a newer feature of most MDM software that creates an encrypted "container" to hold and quarantine corporate data separately from the user's personal data. MDM policies apply only to that container and not the rest of the device.
Caution: You need skilled personnel to implement containerization properly. Without expertise, you get bleed-over, spillage, and an even bigger nightmare than BYOD alone.

Other MDM Capabilities

  • Application Wrapping: Policies that allow employees to safely download apps, typically from an internal store. Can control authentication requirements and whether app data can be stored on the device.
  • Remote Wiping: Instructions sent remotely to erase data on a lost/stolen device. Apple devices have two sides (personal and corporate) — a remote wipe only erases the corporate side. Consent to remote wipe should be required for all BYOD/COPE users.
  • Configuration Profiles & Payloads: Control device settings (passcode, Wi-Fi passwords, VPN configs). Individual settings called payloads can be organized into categories.
  • Conditional Access: Policies that control access to corporate data based on user, location, device state, application sensitivity, and real-time risk assessment.

SCEP (Simple Certificate Enrollment Protocol)

SCEP provisions certificates to network and mobile devices. Since it has no built-in authentication of the requester, two authorization mechanisms exist:

ManualRequester waits for CA operator to approveMore secure but slower
Pre-shared SecretChallenge password delivered out-of-bandUsed for authorization to submit request, NOT device authentication
📋 2. NIST SP 800-163 — Mobile App Vetting
Professor Duke: "NIST SP 800-163 is a GUIDELINE, not a standard. You might want to remember that for the final."

NIST SP 800-163 Rev 1, Vetting the Security of Mobile Applications, helps organizations ensure that mobile apps deployed on organizational devices meet security requirements. Not every app works the same with all devices, so organizations must have a formal vetting process.

The 5 Guidelines

  1. Understand the process for vetting the security of mobile applications
  2. Plan for the implementation of an app vetting process
  3. Develop app security requirements — specify how data should be secured, the deployment environment, and acceptable risk levels
  4. Understand the types of app vulnerabilities and the testing methods used to detect them
  5. Determine whether an app is acceptable for deployment on the organization's mobile devices

App Vetting Process Flow

🕸️ 3. Web Application Attacks

Despite all efforts to design secure web architectures, attacks against web-based systems still occur. Understanding these attack types is critical for both prevention and incident response.

Maintenance Hooks (Slide 17)

Definition: A maintenance hook is a set of instructions built into code that allows someone who knows about the backdoor to connect, view, and edit the code without using normal access controls. Often placed by vendors for customer support and never removed.

Countermeasures: Use host-based IDS to record access attempts, encrypt all sensitive information, implement auditing. Best solution: vendor removes ALL hooks before production. Code reviews should identify and remove them.

TOCTOU — Time-of-Check/Time-of-Use (Slide 19)

TOCTOU attacks exploit the sequence of events that occurs as a system completes common tasks. The attacker inserts themselves between events in multiprocessing systems to introduce changes and alter outcomes.

Fill-in-the-Blank — Will Be on the Exam TOCTOU = Race Condition

Countermeasure: Make critical sets of instructions atomic — they either execute in order and in entirety, or the changes are rolled back.

Cross-Site Request Forgery — CSRF (Slide 20)

CSRF causes an end user to execute unwanted actions on a web application in which they are currently authenticated. Unlike XSS, in CSRF the attacker exploits the website's trust of the browser, rather than the other way around. The website thinks the request came from the user's browser and was actually made by the user.

Key Distinction: XSS exploits user's trust of website. CSRF exploits website's trust of browser. They are opposite attack directions.

Prevention: Web applications should come from a vetted image — one solid image before deploying across systems.

Clickjacking (Slide 22)

A clickjacking attack places a transparent page or frame over a legitimate-looking page. When users click, they think they're clicking on the real site, but they're actually interacting with the attacker's invisible layer.

Caution: Eastern Europe was historically known for clickjacking campaigns. USAA insurance and even the White House website were victims. Social engineering tactics entice users to click. The attacks are being recycled — old attacks come back.
Professor Duke: "What should you be paying attention to instead of caught up in the graphics? THE URL. Always check the URL. My wife and kids, I have to correct them all the time."

Professor Duke: "People buy new phones and jailbreak them — they manipulate the firmware. Then they turn their phones in and don't get any money back because they broke the warranty."

Embedded Systems (Slide 25)

Definition: An embedded system is a computer system with a dedicated function within a larger device. Because they are placed without input from a security professional, security is typically not built in.

Example: Your smartphone has an embedded identity chip placed by Apple or Android — they don't ask you. The government plans to eventually turn your phone into an ID card using these chips.

Firmware Attacks — A New Frontier

Warning: Firmware vulnerabilities were copy-pasted between manufacturers, creating known vulnerabilities across many router brands. Companies cut and pasted firmware coding from other manufacturers, which spread vulnerabilities. Register your router with the manufacturer for push notifications about firmware updates.

System on Chip (SoC): An integrated circuit that includes all components. If designed properly, SoCs allow for stronger battery life and more efficient operation.

🔄 5. SDLC & DevSecOps
Professor Duke: "Know the 8 SDLC steps. You will see a MATCHING question on the exam."

The Software Development Life Cycle is a formal process that ensures security is considered at every phase. When software is developed, the earlier security is considered, the less it will cost. Secure coding standards reduce the attack surface of an application.

SDLC Integration (Slide 37)

Key Concept: The state of the application is determined in every phase of its lifecycle. All possible operations must be identified to ensure the software never enters an insecure state. If it fails security checks at any phase — STOP. Do not proceed.

Zero-Day Vulnerabilities (Slide 40)

Definition: A vulnerability discovered in a live environment for which no current fix or patch exists. Response: isolate and segregate the affected systems immediately.

🔍 6. Code Review & Secure Coding Practices
Fill-in-the-Blank (Slide 50) "_____ is the systematic investigation of the code" = CODE REVIEW

Code review is one of the most critical activities in software assurance. It can take many forms and helps identify both security vulnerabilities and functional issues before software reaches production.

Two Types of Code Review

TypeDescriptionSub-TypesTrade-off
Formal ReviewLine-by-line inspection by multiple people in multiple phases—Most thorough but most time-consuming
Lightweight ReviewMore cursory, done as part of normal developmentPair programming, email review, over-the-shoulder, tool-assisted (most efficient)Faster but less thorough

Input Validation (Slide 55)

Input validation checks all input for issues such as proper format and proper length. It is one of the most important defenses against web application attacks.

MethodHow It WorksRisk
BlacklistingBlocks known bad characters or patternsCan accidentally block legitimate requests
WhitelistingAllows only approved characters or patternsMore restrictive but more secure
Tool Categories: Cloud-based tools, open source tools, and proprietary commercial products. Know these three categories for the exam.

This includes managing desktop sessions, remote sessions, cloud sessions, and VLAN sessions. The answer is given to you in the question — "managing" and "sessions" tells you it's session management.

🔐 7. Authentication — Context-Based & Behavioral

Context-Based Authentication (Slide 59)

Context-based authentication takes multiple factors or attributes into consideration before authenticating and authorizing an entity. It goes beyond simple passwords to evaluate the full context of each access attempt.

Three Key Benefits:
  1. Helps prevent account takeovers made possible by simple password systems
  2. Helps prevent attacks made possible by personal mobile devices lacking security
  3. Helps prevent attacks made possible by the user's location

Behavioral Analysis (Slide 63)

Authentication systems can track the behavior of an individual over time and use this information to detect when an entity performs actions that differ from their normal activity — even if those actions are within the entity's rights.

The Snowden Story: Edward Snowden exploited NSA's failure to practice what they preached. Had NSA implemented behavioral analysis, they would have caught Snowden's abnormal data access patterns much sooner. 32 people were fired as a result of the security failures that enabled Snowden.
