🏗️ 1. What is Cybersecurity Architecture?
KEY IDEA: Cybersecurity architecture is strategic planning — deciding WHERE and WHY to put security controls. NOT just buying tools.

Definition: Strategic planning of security measures across an organization. Think of it as a blueprint for how security controls, processes, and technologies work together. The key word is "strategic" — not engineering.

Architecture ≠ Buying Tools
Thoughtful design of how security operates. Organizations spend millions on best-of-breed tools but without thoughtful architecture integration, those tools sit isolated and ineffective.

Architecture vs. Engineering

Architecture (Strategic)
Focus: WHY/WHAT
Timeline: 3-5 years
Scope: Organization-wide
Output: Blueprints, frameworks
Example: "We need Zero Trust"
Engineering (Tactical)
Focus: HOW
Timeline: Short-term sprints
Scope: Specific systems
Output: Code, configs, deployments
Example: "Configure Azure AD conditional access"
Both Essential
Architecture without engineering = just theory.
Engineering without architecture = chaos.
Feedback flows BOTH ways.
Professor's Tip: Think of an architect as a city planner creating a blueprint. They decide where the roads go, where utilities run, how neighborhoods connect. Engineers then build those roads and utilities according to the plan. One without the other fails.
🍞 2. "Bake In, Don't Bolt On"
Remember This
"Bake In, Don't Bolt On"

Security must be integrated from the design phase forward. Bolting on security after development is expensive, ineffective, and fragile.

Real-World Example: Healthcare Organization

Bolting On (After Development):
Required HIPAA compliance requirements
Result: Rewrite 40% of code
Cost: $2 million
Timeline: 9 months

If Designed In From Start:
Result: Only 15% added development cost
Timeline: No delay
The Math is Clear: Designing security early is orders of magnitude cheaper than retrofitting. Every architecture decision should ask: "What security implications does this have?"
🔐 3. CIA Triad — Foundation of Security
The Three Pillars
C = Confidentiality · I = Integrity · A = Availability

The CIA Triad is the fundamental framework for security objectives. All security measures map to one of these three pillars.

The Three Components

Component Definition Controls Example
Confidentiality Protection from unauthorized access. Only authorized parties can view data. Encryption, least privilege, data classification Patient medical records, trade secrets
Integrity Data must be accurate and cannot be modified except by authorized parties. Digital signatures, hashing, version control Financial transactions, audit logs
Availability Resources must be accessible when needed by authorized users. Uptime, redundancy, recovery plans ER systems, 911 services, e-commerce sites

Key Insight: Trade-offs Exist

The CIA pillars are complementary, but they create tensions when optimized independently:

Confidentiality vs. Availability

Heavy encryption = slower access. Banking apps encrypt everything → users wait for decryption.

Availability vs. Confidentiality

More access points for users = larger attack surface. Servers in 20 countries for availability = harder to secure all of them.

CIA Priority by Industry

Industry Priority Order Reasoning
Military Confidentiality first Secrets are strategic assets
Financial Integrity first Accurate transactions are everything
Emergency Services (911, ER) Availability first Lives depend on system uptime
Healthcare (Holistic) All balanced Patient privacy, accurate records, and system uptime all critical
Professor's Exercise: Credit card payment system. Confidentiality is #1 (PCI-DSS demands encryption and token protection). Integrity is #2 (wrong amount charged = fraud). Availability is #3 (downtime is bad but rare). Remember: "Balance, not absolutes."
Critical Concept: You cannot optimize all three simultaneously. Architecture means making deliberate choices about which matters most IN YOUR CONTEXT.
❓ Six Critical Questions for Security Architects

Before designing any security architecture, the architect must answer these six critical questions that frame the entire effort:

# Question What It Determines
1 What are we protecting? Assets — data, systems, intellectual property, reputation
2 Who are we protecting it from? Threat actors — nation-states, insiders, criminals, hacktivists
3 What are the risks? Likelihood × Impact — risk assessment and prioritization
4 How much security is enough? Risk appetite, ASARP — balance between protection and usability/cost
5 What constraints exist? Budget, timeline, staffing, regulatory requirements, existing tech stack
6 How do we measure success? KPIs, metrics — MTTD, MTTR, compliance scores, incident rates
Prof. Le's Point: "If you can't answer all six questions, you're not ready to start designing. Most failed architectures skip questions 4 and 5."
⚖️ 4. ASARP — As Secure As Reasonably Practicable
Remember This
"As Secure As Reasonably Practicable"

Definition (NIST SP 800-160 V1): The point where investing more in security produces disproportionate negative impact on cost, schedule, or system performance.

Core Principle: Perfect security is impossible. Your job is to find the right balance for your context. Security enables operations; it doesn't stop them.

Context Matters: Examples

Context Priority Sensitivity Security Level Rationale
Medical IoT (infusion pump) Availability > Confidentiality Moderate Patient safety first; breach is bad but patient death is worse
Nuclear facility SCADA Everything maximum Maximum security Availability (reactor down), Integrity (wrong control = meltdown), Confidentiality (weapon data)
Consumer web app Balanced Industry standard Competitive; can't sacrifice UX entirely for security
Super-critical data (nuclear launch codes) Confidentiality absolute Maximum security Cost/schedule/performance are secondary concerns
How to Evaluate ASARP: (1) Determine data sensitivity level. (2) Conduct threat modeling. (3) Assess user access needs and workflows. (4) Calculate cost/usability impact of each control. (5) Find the sweet spot where adding more security breaks the system or workflow.
🔄 5. Inside-Out vs. Outside-In Thinking
KEY IDEA: Both inside-out (building secure features) and outside-in (defending against attacks) are essential. Missing either one leaves you vulnerable.

Two complementary perspectives on security architecture. Both are essential; ignoring one creates blind spots.

Inside-Out (Security Functionality)

Starting Question: What features do we need to build secure systems?

User authentication (who are you?)
Authorization (what can you access?)
Logging and audit trails
Secrets management (API keys, passwords)
API security (rate limiting, validation)
Compliance audit capabilities

Outside-In (Threat Mitigation)

Starting Question: How would an attacker target us, and how do we prevent that?

Input validation (prevent SQL injection, XSS)
Race condition prevention (Time-of-check-Time-of-use)
Secure business logic (can user modify their own ID in URLs?)
Auth bypass prevention
Secure error handling (don't leak info — "invalid login" not "invalid password for user admin")
Injection protection (command injection, etc.)
Real Attack Scenario: System implements MFA (inside-out thinking) but forgets session hijacking prevention (outside-in thinking). Attacker steals session cookie after login, completely bypasses MFA. Both perspectives needed!
Remember: Inside-out = proper features. Outside-in = resilience against attacks. Your architecture needs BOTH.
📋 6. Governance Hierarchy

Definition: The formal process ensuring organizational resources (people, money, time, technology) support the organization's mission and strategic goals.

The Four Levels

Level Formality Owner Scope Example
Policy High Level Executives "We will protect customer data" Data Protection Policy signed by CISO
Standards Prescriptive Subject Matter Experts "Encryption standard is AES-256" Cryptography Standard document
Procedures Step-by-Step Process Owners "How to encrypt a database: [1] [2] [3]" Database Encryption Runbook
Guidance Non-Prescriptive Consultants/Advisors "Consider using salted hashing for passwords" Developer Security Best Practices Guide
Architect's Critical Insight: You must understand WHY before you design HOW. Read the policies to understand what executives care about. Translate business drivers into security architecture. Governance docs reveal what your organization truly values and its risk tolerance.
Flow Direction: Policy drives Standards, Standards drive Procedures, Procedures are enhanced by Guidance. If you see a security control implemented, trace it back to the policy driver.
🔍 7. Root Cause Analysis — The 7 Whys Technique

A technique to dig past surface-level requirements to find the real strategic objective. Ask "Why?" repeatedly until you reach the organization's core mission.

Example: Background Checks

1. Why do we require background checks?
→ To verify employees are trustworthy.

2. Why does trustworthiness matter?
→ Company is liable if we hire someone who commits crimes.

3. Why is liability bad?
→ Damages reputation and customer trust.

4. Why does customer trust matter?
→ Customers feel safe doing business with us.

5. Why does customer feeling safe matter?
→ They spend money with us, not competitors.

6. Why does customer spending matter?
→ Revenue and profit.

7. Why does profit matter?
→ Shareholder value and organizational survival.

The Point: Background checks aren't security theater — they drive customer confidence → revenue. Understanding the true "why" prevents building solutions that satisfy immediate needs but miss strategic objectives.
📊 8. Four Dimensions of Solution Success

Every security solution must be evaluated across four dimensions. Optimizing one at the expense of others is an architectural failure.

The Four Dimensions

1. Effectiveness

Does the solution actually work? Is it fit for purpose?

Example: WPA2 encryption vs. WEP encryption. Both are encryption, but WPA2 is vastly more effective at protecting WiFi. (WEP is broken.)

2. Maturity (CMMI Levels)

How organized and repeatable is the implementation?

Level Maturity Example
1 Ad-hoc (no process) Individual engineer patches servers whenever they remember
2 Managed (documented process) Documented patching schedule, someone owns it
3 Defined (standardized) Patching policy applies org-wide consistently
4 Quantitatively Managed (measurable) We track patch deployment time, success rate, uptime
5 Optimizing (continuous improvement) Automated patching + continuous monitoring + predictive alerts

3. Efficiency

Cost-effectiveness. What's the opportunity cost of this solution?

Budget: $500K + 3 engineers

Option A: Enterprise SIEM for $500K consuming all 3 engineers full-time

Option B: Basic EDR for $200K consuming 1 engineer, freeing $300K + 2 engineers for other security initiatives

Key insight: "Ask not just what a solution costs, but what else we could do with those resources."

4. Alignment

Does the solution fit with organizational culture and team skills?

Even a technically perfect solution fails if your team can't maintain it. A complex SIEM might be effective but if your SOC lacks expertise, it won't stay aligned with your operations.

Architectural Decision Rule: Always evaluate solutions against all four dimensions. If one dimension is failing, the solution will eventually collapse, no matter how strong the other three are.
🧠 9. Optimism Bias

Definition: A cognitive bias where humans believe negative outcomes won't happen to them personally, even though they know the outcomes happen to others.

Example Manifestations in Security

"Our developers write secure code. Other companies get breached, not us."

"We're too small to be targeted."

"Our firewall is good enough; we don't need zero-trust."

Impact on Architecture

Underestimate likelihood of attacks
Underfund security controls
Deprioritize security against other projects
Assume controls work better than they actually do

How to Compensate

Assume Breach Mentality: Design for when breach happens, not if.
Threat Modeling: Use MITRE ATT&CK to systematically map realistic attacks.
Red Team Exercises: Actually test your defenses against competent adversaries.
Plan for Detection/Response: You will be breached. Can you detect it? Can you respond?
Professor's Truth: "Everyone thinks they're special. Everyone is wrong. Assume breach."
⬆️ 10. Top-Down vs. Bottom-Up Architecture

Two approaches to designing security architecture. Neither is complete alone; best practice uses both in concert.

Top-Down Approach

Flow: Org Goals → Tech Goals → Security Requirements → Implementation

Pros:

Strategic alignment with business objectives
Ensures security serves organizational mission

Cons:

Time-consuming to gather all requirements
May miss practical technical constraints

Bottom-Up Approach

Flow: Current Tech Environments → Modular Architectures → Aggregate to Org Patterns

Pros:

Grounded in practical reality
Leverages existing systems and expertise

Cons:

May lose strategic view
Risk of fragmented, inconsistent architecture
Best Practice: Concerted use of BOTH. Start top-down for strategic alignment, verify feasibility with bottom-up technical constraints, iterate.
🎯 11. MITRE Corporation & ATT&CK Framework

MITRE Corporation: A Federally Funded Research and Development Center (FFRDC), non-profit, vendor-neutral. Trusted to conduct research in national interest.

ATT&CK Framework Overview

ATT&CK = Adversarial Tactics, Techniques, and Common Knowledge

700+ techniques documented
14+ tactics organized in kill chain
Based on real-world observations from incidents and threat intelligence
Vendor-neutral (applies to all environments)

Kill Chain (Attack Progression)

# Stage Description Example
1 Reconnaissance Gather info about target Research LinkedIn for employees
2 Initial Access Get first foothold Phishing email with malware
3 Execution Run attacker code Execute macro in Office document
4 Persistence Maintain access if rebooted Create hidden user account
5 Privilege Escalation Gain higher privileges Exploit kernel vuln for admin
6 Defense Evasion Avoid detection Disable Windows Defender
7 Credential Access Steal user credentials Keylogger, credential dumping
8 Discovery Map network and systems ipconfig, net view commands
9 Lateral Movement Move to other systems Pass-the-hash attacks
10 Collection Gather target data Copy files to staging area
11 Exfiltration Extract data Upload to attacker server
12 Impact Achieve objective Ransomware encryption, deletion

ATT&CK Matrices

Matrix Coverage
Enterprise Windows, macOS, Linux, Cloud platforms, Network infrastructure
Mobile Android, iOS
ICS Industrial Control Systems (power grids, manufacturing, etc.)

ATT&CK Navigator

Interactive tool for gap analysis. Color-code your organization's defenses:

RED: No coverage against this technique

YELLOW: Partial coverage

GREEN: Good coverage

Identify your weakest defenses and prioritize remediation.

Critical Statistic

81% of successful intrusions involve ZERO malware.

Attackers use legitimate tools and stolen credentials. This means:
Antivirus/anti-malware alone insufficient
Behavioral detection critical (unusual login patterns, lateral movement)
Credential security is foundation (MFA, credential hygiene)

APT Examples

APT Group Attribution Known Tactics
APT28 (Fancy Bear) Russian GRU (military intelligence) Spear-phishing, 0-days, persistence
APT29 (Cozy Bear) Russian SVR (foreign intelligence) Sophisticated supply chain attacks, detection evasion
Scattered Spider Criminal gang (extortion group) Social engineering, credential theft, ransomware deployment
Why This Matters for Architecture: ATT&CK provides a common language between defenders and executives. Instead of "we need better security," say "We have gaps in lateral movement detection (MITRE ATT&CK technique T1021). Here's our remediation plan."
🛡️ 12. D3FEND, MITRE Engage & Ecosystem

D3FEND Framework

D3FEND = Detection, Denial, and Disruption Framework Empowering Network Defense

Maps defensive techniques that counter ATT&CK techniques. If ATT&CK is "what can attackers do?", D3FEND is "what can defenders do to counter it?"

D3FEND Categories:
Model: Understand systems and data flows
Harden: Strengthen systems against attack
Detect: Identify intrusions in progress
Isolate: Contain compromise
Deceive: Mislead attackers
Evict: Remove attackers from network

MITRE Engage Framework

Active defense through deception. Instead of only defending passively, actively deceive attackers.

Techniques:

Honeypots (fake systems that log attacker activity)
Fake credentials (canary tokens that alert if used)
Deception infrastructure (fake databases, servers)

Goals:

Detect & Expose attacker presence
Impose costs on attacker (consume their time)
Gather intelligence on attacker TTPs (tactics, techniques, procedures)

Phased Implementation Approach

Phase Activity Outcome
1. Assessment Map current defenses against ATT&CK/D3FEND Gap analysis showing weak areas
2. Telemetry Implement logging/monitoring of gaps Visibility into what's happening
3. Detection Engineering Build detection rules for attacks Automated alerting on suspicious behavior
4. Operationalization Integrate into SOC, runbooks, response Effective incident response capability
MITRE Ecosystem Key Insights:
Common Language (ATT&CK): Defenders and executives speak same vocabulary
Evidence-Based: Built on real incident data, not theories
Vendor-Neutral: Applies regardless of what tools you use
Comprehensive Coverage: 14+ tactics, 700+ techniques
Integration Ready: Works with existing tools and processes
⚖️ 13. Regulatory Frameworks

Regulatory compliance is NOT optional. Non-compliance results in fines, loss of operating license, lawsuits, and criminal penalties. These become non-negotiable architectural requirements.

Major Regulatory Frameworks

Framework Scope Key Requirement Penalty for Breach
ISO 27001 Any organization Information security management system Loss of certification, competitive disadvantage
NIST 800-53 U.S. Federal agencies Security controls framework Contract termination, funding loss
HIPAA Healthcare providers Patient data protection, 60-day breach notification Up to $50K per violation, loss of Medicare/Medicaid
PCI-DSS Credit card processors Payment card security, tokenization Up to $100K per month, ability to process cards revoked
GDPR EU citizens' data Data protection, 72-hour breach notification Up to 4% of global revenue, massive reputational damage
SOX Public companies Financial controls and audit trail Criminal penalties, CEO/CFO personal liability
Critical: Regulatory frameworks are not constraints on architecture — they ARE your architecture drivers. Security architects must understand regulatory landscape in their industry and build it into baseline requirements.
Case Study Reality: Healthcare organization: HIPAA 60-day notification requirement means you MUST have: (1) breach detection capability, (2) incident response team, (3) forensic investigation process, (4) legal/compliance team coordinated. These aren't nice-to-haves; they're requirements built into architecture.
🏰 14. Security Zones in Practice

Organize infrastructure into zones of trust. Apply controls at boundaries between zones. Limit what traffic can cross boundaries.

Five-Zone Model (Most to Least Trusted)

Zone Contents Trust Level Access Control
1. Core Assets Crown jewels: databases, secrets, critical systems Highest Extreme restrictions; minimum necessary access only
2. Trusted Zone Enterprise LAN, company-owned devices High Restricted; behind firewall; on VPN required
3. Selective Zone Approved clouds (.mil, .gov), authorized SaaS Medium Controlled; specific data types allowed; contracts required
4. Partner Networks VPN, third-party vendors, contractors Low Restricted to specific functions; monitored; limited to minimum necessary
5. Untrusted Zone Public internet, random users Lowest Maximum restrictions; assume hostile; validate everything
Zone Control Implementation:
Controls enforced at BOUNDARIES between zones
Firewall rules limit cross-zone traffic
Proxies inspect traffic at boundaries
Logging required for all cross-zone attempts

Shadow IT Risk

Shadow IT bypasses zone controls.
Employee uses personal Dropbox account to store company data → data leaves trusted zone into Selective/Partner zone uncontrolled. Your beautiful zone architecture becomes Swiss cheese.
Architectural Implication: Don't just design zones; design policies and monitoring to ensure zones are respected. The architecture is only as strong as its enforcement.
🔌 15. Air Gap Architecture

Definition: Physical or logical disconnection of a system from untrusted networks. Data transfer only through controlled manual processes (no automatic connections).

Key Distinction

Air-Gapped: System has no network connection. Data transfer is manual (USB, written media, strictly controlled).

NOT Air-Gapped: System is disconnected during normal operation but has automatic scheduled sync → This is NOT air gapped; it's just intermittently connected.

Data Transfer Methods

Method Name Description Risk Level
USB drive, removable media Sneakernet Physical media physically moved between systems Medium (media can be infected)
One-way electrical connection Data Diode Hardware device allowing only one-way data flow Low (direction enforced electrically)
Manual inspection, copying Human Review Person manually inspects and copies data Medium (depends on human error)

Use Cases

Nuclear facilities: Control systems completely isolated from internet
SIPRNET (Secret Internet Protocol Router Network): U.S. military classified network
Power grid control: SCADA systems air-gapped from IT networks
Research labs: Isolate sensitive research systems

Important Warning

Even air gaps can be bridged.

Stuxnet Case Study: Iranian nuclear enrichment facility used air-gapped SCADA network. Stuxnet bypassed air gap by: (1) Infecting USB drives used by maintenance personnel, (2) Spreading through Windows systems used for configuration, (3) Reaching isolated SCADA network. Result: Physical damage to centrifuges.

Lesson: Air gaps protect against MOST threats but require defense-in-depth. Assume sophisticated adversary can bridge gaps. Harden endpoints at gap boundaries.
Architectural Decision: Air gaps are expensive (no remote management, slow incident response) and not foolproof (sophisticated adversaries can bridge them). Use only when confidentiality is absolute priority and threat model includes sophisticated nation-state actors.
