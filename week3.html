Week 3 — Network Security Architecture

SEAS8405 · Spring 2026 · Prof. Wayne Le · GWU
CIA in Network Architecture
KEY IDEA: CIA (Confidentiality, Integrity, Availability) applies to network architecture — layer multiple techniques so that compromise of one layer doesn't mean total loss.

Confidentiality in Networks:

  • Encrypted protocols: SSH, TLS, IPsec, S/MIME, VPNs
  • Network segmentation: VLANs
  • Secure tunneling
  • Access control: RBAC, least privilege, JIT (Just-In-Time) access
We are layering multiple techniques so that one still protects us.

Integrity in Networks:

  • Transmission integrity: Checksums and cryptographic verification. Sender calculates hash, receiver recalculates to verify.
  • Storage integrity: Data at rest, digital signatures to detect unauthorized changes
  • Blockchain strategies: Distributed ledger, tamper-evident, excellent for audit logs

Key insight: Network segmentation reduces the number of people with access, which inherently protects integrity.

Availability in Networks:

  • Redundancy + load balancing
  • DDoS protection
  • Multiple paths in topology with multiple routes between nodes
  • Backup and recovery with tested restore procedures

3-2-1 Rule: Maintain 3 copies of data, on 2 types of media, with 1 copy stored offsite.

Always test restore procedure.
Defense in Depth
KEY IDEA: Multiple overlapping security layers mean breaching one doesn't mean total loss — each layer compensates for the weaknesses of others and contains attacks.

Multiple layers of security ensure that if one layer is compromised, the attack is contained. Each layer compensates for weaknesses of the others.

  • When breaching the perimeter, attackers must defeat multiple controls
  • Layers should have clear trust boundaries
  • Not choosing one technique, but layering multiple techniques
Remember This
"One breach ≠ total loss"
If someone breaches your perimeter it doesn't mean total loss.
Design for Resilience
KEY IDEA: Resilience means resources remain available AND protected despite disasters and attacks — it's beyond just availability.

Resources remain available and protected despite unexpected events: natural disasters, man-made disasters, and attacker activity.

Goes BEYOND availability:

  • High Availability: Ensure services remain available during disasters. Implement redundancy in different locations with geographic distribution.
  • Resistance to Attack: Degree to which countermeasures mitigate threat agents. Deploy IDS, integrate threat intelligence.
  • Protected Mechanisms: Security mechanisms themselves are resilient. The vault doesn't unlock when the fire alarm is pulled. Security controls don't fail open.
Prof. Le's Tip: "Protected mechanisms" is the most overlooked aspect — ensure your security controls themselves are resilient and never fail into an open/permissive state during stress.
The mechanism that provides security must remain resilient even when everything else is going sideways.
Network Stack: OSI Model — Security at All Layers
KEY: Most practitioners only think about security at Session and Physical layers. A proper architect applies defense in depth across ALL 7 OSI layers.

Security Controls at Each OSI Layer

  • Application Layer: HTTP, FTP, SMTP, DNS — web firewalls
  • Presentation Layer: Data formatting, encryption, compression
  • Session Layer: Session management, tokens, authentication management
  • Transport Layer: TCP, UDP — end-to-end communication
  • Network Layer: IP, routing, IPsec, network firewalls
  • Data Link Layer: MAC addresses, switches, VLAN segmentation, 802.1X authentication
  • Physical Layer: Cables, electrical signals, physical infrastructure, shielded cables
Most people think of security at Session and Physical. As architect, think across ALL layers.
Network Zones (Review + Expansion)
KEY IDEA: Network segmentation via zones creates trust boundaries — stricter controls apply when traffic crosses from less-trusted to more-trusted zones.

Zone Types & Trust Hierarchy

Key principles:

  • Zone boundaries enforce security controls
  • Traffic from less-trusted to more-trusted zones requires strict inspection
  • Firewalls are placed at zone boundaries

IDS vs IPS

IDS vs IPS Comparison

SystemResponseApproachAnalogy
IDSAlerts only (passive)Detects intrusionsSecurity camera — watches and reports
IPSAlerts + Blocks (active)Detects AND preventsSecurity guard — watches, reports, and stops threats
KEY: Both IDS and IPS use signature-based detection (known attack patterns) and/or behavior-based detection (anomalies) to identify intrusions.
Best Practice: Deploy IPS inline for critical network segments where you can block attacks in real-time; use IDS in monitoring mode elsewhere for alert accuracy.
Redundancy vs Diversity

Redundancy vs Diversity

ApproachDefinitionDefends AgainstExample
RedundancyIdentical components from same vendorIndividual component failureTwo identical Cisco firewalls, RAID drives, redundant power supplies
DiversityDifferent vendors or implementationsSystematic failures and zero-day exploitsCisco + Palo Alto firewalls, AES + RSA encryption, multiple OS types
Remember This
"Redundancy vs Diversity: SAME vs DIFFERENT"
KEY IDEA: Critical systems need BOTH redundancy (handles individual failures) AND diversity (handles systematic failures). One without the other leaves you vulnerable.
Requirements and Design Phase
KEY IDEA: Security decisions made in requirements and design phases are exponentially cheaper to implement than fixing them later — invest upfront for massive ROI.

Security Requirements: Capture authentication, data protection, integrity, and availability explicitly. Requirements must be measurable and follow SMART criteria.

Misuse Cases: Consider how an application might be attacked alongside normal use cases.

Misuse Case Examples

  • Example 1 — Password Reset:
    • Use case: password reset via email link
    • Misuse: attacker requests reset, intercepts email
    • Design: expire links in 15 minutes, require additional verification
  • Example 2 — Profile Picture Upload:
    • Use case: profile picture upload
    • Misuse: upload script disguised as image
    • Design: validate file type, scan for malware, store outside webroot

Threat Modeling: Use STRIDE methodology and MITRE ATT&CK to map attack trees.

Prof. Le's Tip: Every single security requirement in the design phase costs ~$1 to implement. Wait until testing and it costs ~$15. Wait until deployment and it costs ~$100+. Do the work upfront when it's cheap.
Do the work upfront when it's cheap. Getting requirements right equals success for the entire lifecycle.
Boehm's Curve — Cost of Fixing Defects
KEY IDEA: Boehm's Curve shows that security defects found late are exponentially more expensive — each phase compounds the cost multiplier.

The later you find a bug, the more expensive it is to fix:

Cost Multiplier by Phase

PhaseCost MultiplierCost CategoryWhy Expensive?
Requirements1x (baseline)LowestJust update documentation
Design~5xLowRedesign architecture
Development~10xModerateRecode features, re-integrate
Testing~15xHighTest suite regression, retesting cycles
Deployment100x+ (extreme)ExtremePatch production, notify users, handle support escalations, manage reputation damage
Remember This
"Early = $1 | Testing = $15 | Production = $100+"
Prof. Le's Tip: Security vulnerabilities follow Boehm's Curve even more sharply than functional bugs — a security model change in production can cost millions in breach response, legal, and reputation damage.
Changes to security model are much more easily and cost effectively addressed early in lifecycle. Early investment equals massive ROI.
