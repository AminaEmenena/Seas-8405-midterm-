WEEK 5 — Application Security Architecture & SDLC

1. Three Phases of Technology Projects
  • Planning: assessing requirements, marshaling resources, budget, milestones. This is where architecture happens. "Where we design what we build."
  • Implementation: making investments, executing tasks, writing code. Where most attention naturally goes — stakeholders push for progress, leadership wants results. "Planning delays results."
  • Maintenance: continual process of maintaining solution over time. Often de-emphasized.

KEY: Organizations naturally feel pressure to move directly to implementation, making it harder to see big picture. Support considerations pushed because they don't feel urgent.

2. Why Organizations De-emphasize Planning
  • Implementation Pressure: new projects = organizational needs, stakeholders push for progress, planning viewed as delaying solutions, "we can't demo a plan"
  • Maintenance Problem: similar de-emphasis, "we'll cross that bridge when we come to it," focus on getting solution to work first
  • Planned vs Unplanned Cities analogy: Manhattan/Chandigarh/Dubai (grid, easy to navigate, even non-natives can navigate) vs Boston/London/Chennai (complex patterns, harder to navigate, poor documentation, knowledge lost when original engineers leave)
3. Benefits of Structured Systems
  • Performance benefits: systems designed with goals perform better from start
  • Operational benefits: future-proofing built into initial design (build for 10k users but scale to 100k)
  • Maintenance benefits: easier to support when built with structured approach
  • Integration benefits: new components deploy more effectively into systematic layout

KEY: "Benefits compound over time. Year 1 may be only 10% better than ad-hoc, but year 5 may be 300% better."

4. Role of the Security Architect
  • Create the Vision: evaluate goals, constraints, requirements, budget, resources. Not just picking technologies — understanding context (risk tolerance, budget, scope). Ex: "transition from perimeter to zero trust in 3 years"
  • Develop the Blueprint: comprehensive plan guiding entire lifecycle. Tools, how they relate, roadmap, metrics.
  • Work with Stakeholders: collaborate with engineers (feasibility), business leaders (business goal), other architects (ecosystem fit)

KEY: "collaboration and communication" — architect explains to stakeholders how project meets goals

5. Scope of Security Architect
  • Infrastructure Focus: firewalls, anti-malware, IDS/IPS, VPN, network segmentation
  • Application Focus: participate in sprint planning, review application security, define security testing requirements, help developers design authentication
  • Enterprise Focus: oversee security across ALL domains — applications, networks, infrastructure. Define enterprise security standards, create reference architecture, strategic planning.

A small org may have one person doing all three. Large org may have many specific architects.

6. Application Security Architect Role
  • Four Phases: Requirements (scope, purpose) → Development (active coding, testing) → Release (deployment, pre/post activities) → Support (post-release, updates, maintenance, bug fixes)
  • DevOps and continuous deployment may have all phases happening simultaneously
  • Architect's role changes per phase: requirements = ensure security designed in, development = heavy design/implementation, release = coordinating deployment, support = responding to issues

KEY: "Security is woven into system, not bolted on at end."

7. Building Security into SDLC
  • Security Design Reviews: incorporate into release process (waterfall gates or agile sprint retrospectives)
  • Security Testing: manual or automated software security testing in pipeline
  • Code Review: peer review focused on identifying vulnerabilities before deployment
  • Developer Training: train on secure coding techniques. Provide reusable security components and libraries. Open Web Application Security Project (OWASP).

KEY: "Cultural shift: security isn't just architect's responsibility — it's responsibility of ALL employees."

8. Modern Development: DevOps and CI/CD
  • Legacy waterfall had clear phases. DevOps blurs/eliminates boundaries between development, release, support.
  • Goals don't change — architect still develops clear security vision, ensures misuse cases considered, designs attack resistance
  • Execution changes — adapts to development model. Automated security testing in pipeline. Integrate security into rapid deployment without slowing it down.
  • Culture and process — architect influences HOW software gets developed: security design reviews, automated testing, code review, developer training.

KEY: "Building security in means ensuring the development process itself engenders security through systematic practices."

9. Security Requirements and Misuse Cases
  • Use Cases: how system is used for legitimate purposes
  • Misuse Cases: how system might be misused by attacker, accidentally, or by circumstance
  • Security corresponds to threat: every security measure should correspond to a specific misuse case
  • Context determines requirements: how system can be misused drives security requirements as much as how it will be used

Examples:

  • Use case: password reset via email → Misuse: attacker intercepts reset link → Solution: expire links in 15 min, additional verification
  • Use case: profile picture upload → Misuse: upload script disguised as image for remote code execution → Solution: validate file type, scan for malware, store outside webroot

Threat modeling: STRIDE methodology, MITRE ATT&CK maps attack trees

10. Boehm's Curve — Cost Fixing Defects
  1. Requirements: lowest cost (just update docs)
  2. Design: 5x cost
  3. Development: 10x cost
  4. Testing: 15x cost
  5. Deployment: 100x+ cost (patch, notify users, support failures, potential exploitation)

Example: Find bug immediately = 1 hour, modify code, unit test, done. Find bug after shipping 1M units = weeks of work, update source, integration test, schedule release, rebuild, notify users, support patch failures.

KEY: "Do the work upfront when it's cheap."
