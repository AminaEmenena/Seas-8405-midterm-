🎯 1. Why Scope Matters
  • Fitness of Purpose: Purpose changes design. Architecture must match organizational purpose and function.
  • Requirements Flow from Scope: Security requirements aren't independent — they logically flow from what is being defended and against what threats. Security measures don't make sense without threats to defend against.
  • Practical Boundaries: No single architect can be responsible for everything. Scope defines bounded areas.
  • KEY: Scope creep is the enemy of project management. Focus on developing, validating, and agreeing on scope before execution.
KEY IDEA: Scope is the foundation of security architecture. Without clear scope, security requirements float in the void—meaningless and unmeasurable.
Remember This
"Scope = What, Threats = Why, Requirements = How"
🔄 2. Scope is Iterative, Not Fixed

Scope changes as organization changes and as design decisions are made. It's deliberately a tool of the architecture team.

Four phases:

  • 1) Initial Scope Setting
  • 2) Requirements Gathering — redefine based on stakeholder input
  • 3) Design Development — adjust based on dependencies/limitations
  • 4) Scope Documentation — finalize and document. If it's not in the statement of work, we don't do it.

Not about perfection initially — close enough to refine with stakeholders.

KEY: Scope is not a one-time event. It evolves as you learn more about the organization, technology, and constraints. Think of it as a living document.
Prof. Le's Tip: The first scope you write will be wrong. Plan for it. Build in buffer and stakeholder time to refine. That's not failure—that's the process.
💡 3. Advantages of Setting Scope
  • Focus: Subset of broader problem = more manageable (like OSI stack focusing one layer at a time)
  • Collaboration: Clearer boundaries allow delegation and sharing
  • Acknowledge Limitations: Areas we cannot affect due to budget/skills can be removed from scope
  • Manage Known Risks: Remove from consideration areas representing known accepted/mitigated risks
Best Practice: Use scope boundaries to explicitly say "this is out" — not because it doesn't matter, but because it's someone else's problem or a known constraint. Document it so no one wonders later.
⚠️ 4. Critical Architectural Risks and Empty Scope

Sometimes scope is so constrained it's effectively zero. Organizational barriers, technical debt.

Warning signs:

  • Overwhelming constraints
  • Technical debt
  • Organizational barriers blocking progress

Solutions:

  • Recognize warning signs early
  • Address root causes first (clear hurdles before architecture)
  • Use initial scope as diagnostic (identify problems early)

A house on quicksand — need to address foundation first. Does not mean organization doesn't need architecture.

KEY IDEA: Empty Scope = architecture impossible due to foundation problems. This isn't failure. It's a diagnostic: go fix the foundation, then come back to architecture.
🏰 5. Enterprise Security Architecture Definition
  • Focuses on administrative, procedural, and technical controls securing enterprise or subset
  • Includes infrastructure, network, systems, operational security
  • Focus on environment — generally cannot directly change software employees use. Focus is on controls, countermeasures, operational constraints around existing apps
  • Systematic approach: Selection, application, operationalization, monitoring of security decisions across enterprise
KEY: Enterprise Security Architecture is about what surrounds and supports applications, not the applications themselves. Build the secure environment; apps live inside it.
📊 6. Organizational and Technology Boundaries

Organizational: Multiple business units responsible for different portions. Example: retail banking on balance viewing, brokerage on portfolio interface. Different teams, different priorities.

Technical: Legacy technologies (mainframes vs modern APIs) require different approaches. Different security expertise needed (mainframe CICS vs RESTful API security).

Key: Scope can be artificially constrained to level of tier, component, or service based on technology or organizational factors.

Best Practice: Map your organization's teams to your scope boundaries. If Team A owns retail and Team B owns brokerage, your scope can reflect that split. Align scope with accountability.
Prof. Le's Tip: Technology boundaries are often your hidden scope leverage. Legacy mainframes and modern APIs have different risk profiles. Use that to set realistic, focused scope.
🔄 7. Modularity and Scope Flexibility
  • Monolithic (Tight Integration): Client-server, same address space. Very difficult to scope out individual components.
  • N-Tier (Loose Integration): UI, application tier, middleware, database. Can scope out database and focus on UI/application.
  • Microservices (Maintenance-style): Stateless components using HTTP transport. Easy to set scope around individual services. Highly flexible and modular.

KEY: REST and microservice architecture make it much easier to carve scope at component or service level.

KEY: Architecture modularity directly enables scope flexibility. Monoliths force you to scope the whole system. Microservices let you pick and choose which services to focus on.
Remember This
"Loose Coupling = Flexible Scope"
⚡ 8. Development Models Impact Scope
  • 1970s-80s Waterfall: Sequential phases, rigid, must get scope right early
  • 1990s Unified Process: Iterative, more flexible, still structured
  • 2000s Agile/Lean: Iterative/incremental, emphasize flexibility, scope can change sprint by sprint
  • 2010s DevOps: Continuous delivery, scope adjusts weekly as new services spin up. Amazon deploying every minute.
KEY IDEA: Your development methodology shapes scope. Waterfall demands perfect scope upfront. Agile/DevOps lets you evolve it. Know your process, set scope accordingly.
Best Practice: For waterfall, invest heavily in scope definition upfront. For agile/DevOps, set initial scope loosely and plan for regular reviews.
📋 9. Four-Step Scope-Setting Process
  • Step 1: Consider High-Level Goals — Review existing documentation, discuss with stakeholders, seek out ALL stakeholders (users, sponsors, executives, security, auditors, counsel, risk managers, compliance, partners)
  • Step 2: Review Contextual Constraints — For application (service boundaries, modularity, technology constraints, organizational boundaries) and enterprise (business/structural boundaries, risk management scope, strategic planning, maturity level)
  • Step 3: Set Initial Scope — Crystallize boundaries. What's in, what's out, what's middle ground. Must include core systems and critical functions.
  • Step 4: Validate and Refine — Stakeholder feedback loop. Go back to stakeholders. Demonstrates listening, catches blind spots, saves time long-term. Don't make decisions in isolation.
KEY: The 4-step process is your roadmap. Follow it in order. Each step informs the next. Skip steps and you'll miss critical stakeholders or constraints.
Remember This
"Goals → Constraints → Boundaries → Validation"
Prof. Le's Tip: Step 4 is where most people cut corners due to time pressure. That's where the real work happens. Invest in validation—it saves rework later.
🎯 10. Key Principles for Scope Success
  • Scope is Iterative: Question assumptions, amend/update, socialize changes
  • Scope is a Tool: Selectively delegate portions to other teams. Don't ignore concerns — address elsewhere or at different time.
  • Avoid Scope Creep: While iterative, uncontrolled creep destroys projects. Freeze scope during execution, changes require agreement and re-validation.
  • Communicate Changes: Document, socialize with stakeholders, get buy-in, update plans.
KEY IDEA: Scope creep is iterative scope without control. It's the difference between evolving a document and watching a project drown in forever-changing requirements.
⚖️ 11. Risk Management as Scope Input
  • ISO 27001/ISMS Scope: Formal document defining what's in/out of security program
  • Formal Risk Frameworks: ISO 31000, NIST RMF, OCTAVE — all include scope definition. Risks at highest priority inform architectural scope.
  • Using Existing Risk Scope: Directly incorporate or reference existing risk management scope.
Best Practice: Don't invent scope from scratch. Check if your organization has an existing risk management scope (ISO 27001, NIST RMF). Use that as your baseline. You'll save time and align with organizational priorities.
KEY: Risk frameworks already define scope. Leverage them. That's the whole point of using a framework.
🧠 12. Scope-Setting Catch-22

The Problem: Until you validate with stakeholders, scope likely falls short. But how do you talk to stakeholders without knowing borders of effort?

Solution:

  • Two-phase approach (initial scope allows error as you set up stakeholder discussions; final scope after validation)
  • Reduced stakes and barriers (initial doesn't need to be perfect)
  • Iterate and refine (output of initial scope WILL change)

Know the rules well so you can break them effectively. Adapt formal methods to practical effect for efficiency.

KEY IDEA: The Catch-22 isn't a problem to solve—it's a feature. It's permission to draft a rough scope, use it to organize stakeholder conversations, then refine. Don't get paralyzed by perfection.
Remember This
"Draft. Discuss. Refine. Repeat."
