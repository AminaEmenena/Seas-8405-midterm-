📋 1. What is an Enterprise Blueprint?
DEFINITION: An enterprise security blueprint is a set of related documents that capture and codify design elements. It serves a dual purpose: capturing the desired future state AND creating buildable, implementable documentation.
  • Framework alignment: TOGAF calls it Architecture Vision, SABSA calls it Conceptual Security Architecture
  • House analogy: Think of a house blueprint — floor plan, electrical diagram, plumbing layout, elevation drawing — all are separate documents related to the same project
  • Not a single document: A blueprint is a collection, not one diagram. Each document captures a different design dimension of the same architecture
Remember This
"Blueprint = many docs, one vision."
💡 2. Why Blueprints Matter (4 Reasons)
KEY IDEA: Blueprints serve four critical functions in enterprise security architecture. Without them, architecture stays in one person's head.

Four Reasons Blueprints Matter

ReasonWhat It MeansWho Benefits
Socialize Design Get buy-in from executives, technical teams, and stakeholders Leadership, cross-functional teams
Implementation Template Framework for network/identity/cloud/app security teams to build from Engineering and operations teams
Progress Baseline Measure against blueprint, track milestones and deviations Project managers, architects
Status Reporting Answer "where are we?" questions for executives Executives, board, auditors
Remember This
"S.I.P.S. — Socialize, Implement, Progress, Status"
📝 3. Blueprint Documentation Principles
KEY IDEA: Documentation is only valuable when it is practical, used, and useful. Minimize documentation — don't create docs nobody reads.
  • Minimize documentation: Only create what will actually be used. Every document should serve a clear purpose
  • Personnel Continuity: People leave organizations. New architects need to pick up where others left off. Documentation bridges that gap
  • Collaboration Enablement: Executives fund it, experts refine it, operations implement it. Documentation is the shared language between these groups
Prof. Le's Tip: "If nobody reads your 200-page architecture document, it's not documentation — it's shelf decoration. Keep it practical."
Best Practice: Ask yourself before creating any document: "Who will read this? What decision does it support? When will it be updated?" If you can't answer these, don't create it.
🏗️ 4. TOGAF Framework
DEFINITION: TOGAF = The Open Group Architecture Framework. It provides a systematic approach to enterprise architecture with extensive documentation and broad industry recognition.

5 Advantages of TOGAF

  1. Extensive Documentation: Comprehensive guidance for every phase
  2. Free to Use: No licensing costs — accessible to any organization
  3. Widely Recognized: Industry standard with broad adoption
  4. Customization Support: Section 4.3 explicitly says "adapt this to your needs"
  5. Systematic Approach: Structured methodology reduces ad-hoc decisions

ADM Phases for Security Architecture

Preliminary → Phase A
Architecture Vision → Phase B
Business Architecture → Phase C
Information Systems → Phase D
Technology Architecture
Important: Compress and adapt TOGAF for security-specific contexts. You don't need every phase at full rigor for security architecture work.
⚖️ 5. Adapting Rigor to Context
KEY IDEA: Large projects need more rigor (validation, stakeholder consensus, governance). Smaller projects prioritize agility. Perfect is the enemy of good enough.

Three Options When Full Rigor is Unattainable

  1. Option 1: No design at all — Just wing it (worst option)
  2. Option 2: Accept risk — Acknowledge gaps while working on the larger design
  3. Option 3: Adjust rigor to fit timeframe — The preferred approach. Maintain defensibility while adapting to constraints
Best Practice: Option 3 is almost always the right choice. Some architecture is better than no architecture. Scale the rigor to match the project's risk and timeline.
Remember This
"Perfect is the enemy of good enough. Ship defensible architecture, not perfect architecture."
⚔️ 6. Kill Chain (Lockheed Martin)
DEFINITION: The Cyber Kill Chain is a framework developed by Lockheed Martin that describes the 7 phases of a cyber attack, from initial reconnaissance to final objective completion.

7 Phases of the Kill Chain

#PhaseDescriptionArchitect's Question
1ReconnaissanceAttacker gathers information about targetWhat information is publicly exposed?
2WeaponizationAttacker creates exploit payloadWhat vulnerabilities exist in our stack?
3DeliveryPayload transmitted to target (email, web, USB)How do we filter inbound threats?
4ExploitationVulnerability triggered, code executesHow do we harden and patch?
5InstallationPersistence established in environmentHow do we detect unauthorized changes?
6Command & ControlAttacker establishes remote control channelHow do we monitor outbound traffic?
7Actions on ObjectivesAttacker achieves goal (exfiltration, destruction)How do we limit blast radius?
Prof. Le's Tip: "For each asset, walk through all 7 kill chain phases. Ask: how could this happen? How would we recognize it? How do we prevent or detect it?"
Remember This
"R-W-D-E-I-C-A: Recon, Weapon, Deliver, Exploit, Install, Command, Act"
🛡️ 7. Assets in Architecture
DEFINITION: An asset is anything important to the organization — data, personnel, intellectual property, business processes, customer trust, and reputation.

Two Paths to Identify Assets

  • Path 1: Systematic Inventory — Catalog everything the organization owns and values
  • Path 2: Derive from Goals — Start with business goals and work backward to identify what supports them

3 Reasons to Document Assets

  1. Verify importance: Confirm that each asset truly matters to the organization
  2. Explain protection: Describe how security architecture protects each asset
  3. Understand threats: Identify what threats target each asset
Healthcare Example: Assets include physical servers + patient data + organizational reputation + 24/7 care delivery capability. Each requires different protection strategies.
📊 8. Assets, Threats & Risk (3 Questions)
KEY IDEA: Security architecture answers three fundamental questions. Everything else is detail.

The Three Questions

QuestionWhat It CoversFramework Tool
Q1: What are we protecting? Assets — identify and prioritize them Asset inventory, business impact analysis
Q2: What bad things can happen? Threats — walk through the kill chain for each asset Kill chain analysis, threat modeling
Q3: What's the result? Risk = Impact x Likelihood Risk matrix, risk register
Important: Focus on high impact + reasonable likelihood. You cannot defend everything equally. Prioritize based on risk, not fear.
Remember This
"Protect → Threat → Risk. Don't defend everything — defend what matters."
🎯 9. From Goals to Capability
KEY IDEA: Architecture must bridge the gap between what the organization wants (goals) and what it can actually do (capabilities). You need both views.
  • Goal view (the "what"): Desired outcomes — what the organization wants to achieve. Connects to business value and executive language
  • Capability view (the "how"): Specific capabilities needed to achieve those goals. Determines feasibility and resource requirements
Best Practice: You need both views. Goals connect to business value (executives care about this). Capabilities determine feasibility (engineers care about this). The architect translates between them.
Caution: Avoid too much detail early in the process. Validate your assumptions about goals and capabilities before investing in detailed design. Getting the wrong answer very precisely is still wrong.
🗺️ 10. Roadmap Best Practices (7 Components)
KEY IDEA: A security roadmap has 7 components. Keep it brief, structure for discussion, not final documentation.

7 Components of a Security Roadmap

  1. Charter/Goal Statement: Why does this roadmap exist? What problem are we solving?
  2. Scope: What's in and what's out? Boundaries matter
  3. Principles: Guiding rules that constrain and focus decisions
  4. Goals & Outcomes: Measurable objectives and success criteria
  5. Transition Milestones: Key checkpoints and phase gates
  6. Project Management Artifacts: Timelines, resource plans, dependencies
  7. Impacts: What changes? Who is affected? What are the risks of the transition?
Prof. Le's Tip: "A roadmap is a conversation starter, not the final answer. Keep it brief enough that executives will actually read it."
📝 11. Two-Phase Documentation
KEY IDEA: Don't create detailed technical documentation until your high-level roadmap is validated. Two phases save time and reduce waste.

Two-Phase Approach

PhaseDocumentAudiencePurpose
Phase 1 High-Level Roadmap Business-facing, executives Abstract, lightweight — get buy-in and validate direction
Phase 2 ADD (Architectural Definition Document) Technical teams, implementers Complete technical package for implementation
Phase 1: Roadmap → Stakeholder Validation → Phase 2: ADD
Critical Rule: Only create Phase 2 (ADD) AFTER the roadmap has been validated by stakeholders. Creating detailed docs before validation wastes time and ensures misalignment.
⚖️ 12. Top-Down vs Bottom-Up & Architectural Principles
KEY IDEA: Top-down is ideal (executive sponsor from start). Bottom-up is reality for many orgs (start small, show value). Hybrid approach works best.

Top-Down vs Bottom-Up

ApproachHow It WorksWhen to Use
Top-Down Executive sponsor from the start, strategic mandate drives architecture Ideal — when you have executive support and budget
Bottom-Up Start small, demonstrate value, build support incrementally Reality for many orgs — when executive buy-in doesn't exist yet
Hybrid Start bottom-up to prove value, then scale to top-down Best of both worlds — pragmatic path to full architecture

4 Components of Architectural Principles (TOGAF)

Principle Structure

  1. Name: A short, memorable title for the principle
  2. Statement: Clear, concise declaration of the principle
  3. Rationale: Why this principle exists — the business justification
  4. Implications: What this principle means in practice — constraints and consequences
Remember This
"N.S.R.I. — Name it, State it, Rationalize it, Implication it"
